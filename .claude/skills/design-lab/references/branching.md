# Branching workflows

The git workflow underpinning everything in the design lab. Read this whenever the designer wants to switch branches, save work, take a draft, rebase, publish, or create a new branch. Every other reference file leans on this one.

The golden rule: **never act silently on a branch-affecting operation.** Always surface the action, translate the term, and ask for confirmation.

## The branch naming convention

```
develop                                              base — never edited directly
└── claude-design-lab                                base — never edited directly
    ├── design/<github-id>/components/<slug>         component work (reusable)
    └── design/<github-id>/prototype/<slug>          prototype work (app integration)
```

- `<github-id>` — the designer's GitHub username. Try `gh api user --jq .login` first; if `gh` isn't authed, infer from `git config user.email` and **always confirm** with the designer.
- `<slug>` — lowercase, hyphens, descriptive, short. Examples: `auth-redesign`, `file-picker-v2`, `onboarding-experiment-a`.

## Creating a new design branch

Before code changes, get the designer onto the right branch.

**1. Decide component or prototype.** Ask the designer if not obvious:

> Is this work going to be a new **component** (a reusable building block — e.g. a button, a list item, a menu) or a **prototype** (a feature mockup wired into the element-web app)? They live on different branches because components can be reused across multiple prototypes.

**2. Get the GitHub ID.** Prefer `gh api user --jq .login`. Fall back to asking. Cache it for the session.

**3. Pick a slug.** Suggest one based on what they described, then confirm.

**4. Run the create flow.** Show the commands you're about to run **before** running them:

> I'll do these things in order:
>   1. Make sure we're up to date with `claude-design-lab`
>   2. Create a new branch called `design/davidlangley/components/auth-button` from it
>   3. Switch you onto the new branch
>
> Shall I proceed?

Then:

```bash
git fetch origin
git checkout claude-design-lab
git pull --rebase origin claude-design-lab       # only if the branch is published; otherwise skip
git checkout -b design/<github-id>/<type>/<slug>
```

After creating, **don't publish (push) yet** — wait until the designer has saved (committed) at least one meaningful change. Then offer to publish.

## Switching branches with unsaved changes

If the designer has uncommitted changes and wants to do anything that requires a clean tree (switch, rebase, pull), **always** offer the three-way choice. Never silently stash, never silently discard.

The phrasing to use:

> You have **2 unsaved changes** (uncommitted) on `design/davidlangley/components/auth-button`. Before we switch, would you like to:
>
>   1. **Save them** (commit) here — best if this is a meaningful checkpoint we should keep
>   2. **Keep them as a draft** (stash) — they'll come back the next time you return to this branch
>   3. **Throw them away** (discard) — only if you're sure you don't need them
>
> Which one? (If you're not sure, **option 2** is the safest — we can always discard later.)

**Save (commit) flow:**

> Great. What should we call this checkpoint? A short message helps you find it later — something like "Add hover state to auth button". (I can suggest one if you'd like.)

Then:

```bash
git add -A
git commit -m "<message>"
```

After committing, optionally offer to publish (push) — see below.

**Draft (stash) flow:**

```bash
git stash push -u -m "<auto-generated label based on what's in the diff, e.g. 'WIP on AuthButton hover'>"
```

Always include a label so the designer can find the draft later. Confirm:

> Saved as a draft. You can bring it back any time by switching to this branch and asking me to "restore my draft".

**Discard flow:** require an extra confirmation because this is destructive.

> Just to double-check — this will permanently throw away the 2 modified files. There's no undo. Are you sure? (yes / no)

Only if yes:

```bash
git restore .
git clean -fd      # only if there are untracked files; ask first
```

## Restoring a draft (stash pop)

When the designer returns to a branch that had a draft, offer to restore it:

```bash
git stash list                                  # check what's there
```

> When you left this branch, you saved a draft: "WIP on AuthButton hover". Would you like to bring it back? (yes / leave it as a draft for now)

If yes:

```bash
git stash pop          # if it applies cleanly
```

If it doesn't apply cleanly, walk the designer through the conflict in plain language ("These two changes touched the same line — which version should we keep?").

## Saving (committing) work

Encourage the designer to save (commit) at meaningful checkpoints. Don't nag, but offer:

> This feels like a good checkpoint — shall I save (commit) what we have so far? Suggested message: "Add empty state to FilePicker stories".

**Commit message guidelines for design work:**
- Imperative present tense: "Add", "Update", "Refactor".
- Mention the component or feature being touched.
- Keep it short — one line is fine. No long body needed for design-lab branches.

```bash
git add -A
git commit -m "<message>"
```

## Publishing (pushing) a branch

After the first save, offer to publish:

> Want me to publish this branch to GitHub (push)? That makes it visible to others and lets you deploy a Netlify preview later.

For first publish:

```bash
git push -u origin <branch-name>
```

For subsequent publishes:

```bash
git push
```

If `git push` would be a fast-forward, just do it. If it would require a force-push (because of a rebase), confirm first:

> A heads up — because you updated this branch with the latest develop earlier (rebase), publishing now requires a **force-push**. That's normal after a rebase, but it overwrites the published history. Anyone else working on this branch would need to reset their copy. Shall I proceed? (yes / no)

```bash
git push --force-with-lease       # always use --force-with-lease, never --force
```

## Keeping a branch up to date (rebase)

Design branches should regularly absorb updates from their parent so they don't drift. Offer to rebase whenever:

- The designer asks "is my branch up to date?"
- The branch is 5+ commits behind its parent
- Before a Netlify deploy
- After the designer returns to a branch they haven't touched in a few days

The rebase flow, always confirm first:

> Your branch is **8 commits behind** `claude-design-lab`. Updating now means you get the latest fixes and won't hit surprises later. I'll do this:
>
>   1. Make sure all your work is saved (committed) or kept as a draft (stashed)
>   2. Pull the latest `claude-design-lab` from GitHub
>   3. Replay your changes on top (rebase)
>
> Shall I proceed?

If there are unsaved changes, run the save/draft/discard dialog first.

```bash
git fetch origin
git rebase origin/claude-design-lab
```

If conflicts come up, walk through them in plain language:

> The rebase paused because two changes touched the same lines in `FilePicker.tsx`. Let me show you both sides:
>
> - The newer version from `claude-design-lab`: …
> - Your version: …
>
> Which would you like to keep? Or want to mix the two?

After a successful rebase, remind the designer that if the branch was already published, the next push needs `--force-with-lease`.

## Switching to an existing branch

When the designer wants to continue work on an existing branch:

1. Run the unsaved-changes dialog if needed.
2. Switch:
   ```bash
   git checkout <branch-name>
   ```
3. Show the state report for the new branch (see [state-and-tree.md](state-and-tree.md)).
4. If there's a draft (stash) attached to this branch, offer to restore it.
5. Ask what they'd like to do next.

## Picking the right branch for the work

If the designer starts editing files and the location doesn't match the branch type, stop them gently:

**On a prototype branch, editing `packages/shared-components/...`:**

> Quick check — you're editing a shared component (`packages/shared-components/src/core/AuthButton/...`), but you're on a **prototype** branch (`design/davidlangley/prototype/new-onboarding`). Components live on their own branch so other prototypes can reuse them. Would you like to:
>
>   1. **Move this change to a component branch** — I'll save your work as a draft, switch to a new component branch off `claude-design-lab`, and bring the change over
>   2. **Continue here** — if this is genuinely one-off and we'll never want to reuse it
>
> Which one?

**On `develop` or `claude-design-lab` directly, editing anything:**

> You're on `claude-design-lab` (or `develop`), which is a base branch — we don't edit it directly because everyone shares it. Let me set you up with a proper design branch first. Is this a **component** or a **prototype**?

## Deleting / cleaning up a branch

Only when the designer explicitly asks to delete a branch. Always confirm and check that work is preserved.

> Just to confirm — you want to delete `design/davidlangley/components/old-experiment`. Before I do:
>
> - Is this branch published (pushed)? If yes, you may want to keep the published copy for reference.
> - Are there any unsaved drafts (stashes) attached to it?
>
> I'd recommend we keep the published copy on GitHub and just delete the local one. Want to do that?

```bash
git branch -d <branch-name>          # safe delete — refuses if unmerged
# only with explicit confirmation:
git branch -D <branch-name>          # force delete
```
